# Development

This guide describes a clean local development setup for contributors and maintainers.

## Prerequisites

- Node.js 20
- npm
- Docker Desktop or Docker Engine
- GitHub CLI, optional but useful for release and PR work
- PostgreSQL client tools, optional for direct database inspection

The CI workflow uses Node.js 20. Local development should use the same major version unless you are intentionally testing runtime compatibility.

## First-Time Setup

Install dependencies:

```bash
npm ci
```

Create local env:

```bash
cp .env.example .env.local
```

Generate a local auth secret and paste it into `NEXTAUTH_SECRET` and `AUTH_SECRET`:

```bash
openssl rand -base64 32
```

Start the database:

```bash
docker compose up -d db
```

Apply schema:

```bash
npm run db:migrate
```

Seed sample data:

```bash
npm run seed
```

Run the app:

```bash
npm run dev
```

Open `http://localhost:3000`.

## Database Work

Schema source:

- `database/schema.ts`

Drizzle config:

- `drizzle.config.ts`

Generated migrations:

- `migrations/`
- `migrations/postgres/`, when generated by the current config
- `migrations/mysql/`, historical migration artifacts

Current application code uses PostgreSQL. Treat PostgreSQL as the canonical production database.

### Apply Schema Locally

```bash
npm run db:migrate
```

This runs `drizzle-kit push` against `DATABASE_URL`.

### Generate Migration Files

```bash
npm run db:generate
```

Review generated SQL before committing.

### Open Drizzle Studio

```bash
npm run db:studio
```

### Use Adminer

Start the full Compose stack or just Adminer and DB:

```bash
docker compose up -d db adminer
```

Open `http://localhost:8080`.

Connection values:

| Field | Value |
| --- | --- |
| System | PostgreSQL |
| Server | `db` |
| Username | `POSTGRES_USER` |
| Password | `POSTGRES_PASSWORD` |
| Database | `POSTGRES_DB` |

## Adding A Feature

Use this flow for most product changes:

1. Read the relevant page, component, action, route, and service files.
2. Confirm whether the flow is student, admin, or public.
3. Add or update validation in `lib/validations.ts` when accepting input.
4. Use server actions for form-first mutations and API routes for JSON integrations or client fetch paths.
5. Revalidate affected cache tags when catalog, borrow, review, or recommendation data changes.
6. Add focused tests for shared logic, action behavior, permissions, or regressions.
7. Update documentation in the same PR when setup, behavior, routes, or operations change.

## Route Ownership

| Change type | Likely owner files |
| --- | --- |
| Student page | `app/(root)/**`, `components/**`, `lib/actions/**` |
| Admin page | `app/admin/**`, `components/admin/**`, `components/Admin*.tsx`, `lib/admin/actions/**` |
| Public JSON read | `app/api/**/route.ts`, service in `lib/services/**` |
| Admin JSON route | `app/api/admin/**/route.ts`, `lib/admin/route-guard.ts` |
| Database schema | `database/schema.ts`, `migrations/**` |
| Cache behavior | `lib/cache/**`, affected mutation files |
| Email or workflow | `lib/workflow.ts`, `lib/services/email-service.ts`, admin automation routes |

## Styling And Design System

Design tokens and global styling live mostly in:

- `app/globals.css`
- `styles/admin.css`
- `tailwind.config.ts`
- `components/ui/**`

Current product direction:

- Academic, institutional, Mundiapolis navy and restrained gold.
- Warm off-white background and white surfaces.
- Serif headings with system sans UI.
- No repeated all-caps micro-labels.
- No nested card-heavy dashboard templates.
- Primary actions should be rare and meaningful.

See `PRODUCT.md` and `DESIGN.md` before making broad UI changes.

## Code Quality Rules

- TypeScript should remain strict enough to catch contract drift.
- Lint runs with `--max-warnings=0`.
- Shared helpers should have tests when behavior is non-trivial.
- Admin routes must guard authorization server-side.
- User-owned data routes must prevent cross-user access.
- Do not rely on client-side hiding for permissions.
- Do not print secrets or sensitive user data in logs.

## Working With Existing Generated Files

Generated artifacts you usually should not commit:

- `.next/`
- `coverage/`
- `artifacts/`
- release archives unless intentionally publishing from the repo
- local env files

Generated artifacts that may be committed when intentionally changed:

- Drizzle migrations
- performance baselines
- documentation updates

## Troubleshooting

### The app cannot connect to the database

Check:

```bash
docker compose ps
```

Then confirm `DATABASE_URL` in `.env.local` uses the host reachable from your process:

- Running app on host machine: `localhost`
- Running app inside Compose: `db`

### Auth redirects loop

Check:

- `NEXTAUTH_SECRET` is set.
- `NEXTAUTH_URL` matches the URL you are opening.
- Browser cookies are not stale from another local environment.

### Image uploads fail

Check:

- `NEXT_PUBLIC_IMAGEKIT_PUBLIC_KEY`
- `NEXT_PUBLIC_IMAGEKIT_URL_ENDPOINT`
- `IMAGEKIT_PRIVATE_KEY`

Then call:

```bash
curl http://localhost:3000/api/auth/imagekit
```

### Rate limit errors locally

Set:

```bash
DISABLE_RATE_LIMIT=true
```

Development also bypasses rate limits when Redis config is missing.

### Seed script fails

Run schema migration first:

```bash
npm run db:migrate
npm run seed
```

If the database is in a bad local state, remove the Docker volume only after confirming there is no data you need:

```bash
docker compose down -v
docker compose up -d db
npm run db:migrate
npm run seed
```
